# Semgrep CI — статический анализ и выгрузка в Svacer

## Описание

Шаблон запускает статический анализ кода с помощью [Semgrep](https://semgrep.dev/), формирует отчёт в формате SARIF вместе с исходным кодом и выгружает его на сервер отчётов Svacer.

**Когда запускается джоба:**

- при открытии/обновлении Merge Request;
- на ветках `release-*`;
- на ветке по умолчанию;
- на тегах вида `v1.2` или `v1.2.3`;
- по расписанию (schedule);
- при ручном запуске (web).

Скрипт ищет в репозитории **все директории**, в которых есть конфиг Semgrep. В каждой такой директории должен быть ровно один файл конфигурации: **`semgrep.yml`** или **`semgrep.yaml`** (расширение может быть разным в разных директориях). Если в одной директории есть оба файла, используется `semgrep.yml`. Для каждой найденной директории выполняется сканирование своего кода, упаковка SARIF и исходников в архив и выгрузка в **отдельный проект** в Svacer (один образ = один проект).

## Что нужно в репозитории

В репозитории может быть **несколько директорий с исходным кодом**, каждая из которых анализируется отдельно и импортируется в Svacer как отдельный проект. В **каждой такой директории** должен лежать один конфиг: файл **`semgrep.yml`** или **`semgrep.yaml`** (в одной директории — один файл, расширение может быть любым из двух).

### Формат конфига (semgrep.yml или semgrep.yaml)

Конфиг лежит в директории с исходным кодом. Обязательные поля:

| Ключ    | Описание |
|--------|----------|
| `image` | Имя образа, например `backend`, `frontend`. Каждая директория с конфигом = один образ. |
| `module` | Имя модуля |
| `path`  | Путь **относительно этой директории** — что сканировать (например `.` или `src`). |

Опционально:

| Ключ | Описание |
|------|----------|
| `svacer_project_group` | Группа проекта в Svacer для этого образа (переопределяет общую настройку). |

**Пример** — в репо две директории с кодом, в каждой свой конфиг:

- `images/backend/semgrep.yml`:

```yaml
image: backend
module: my-service
path: .
```

- `images/frontend/semgrep.yaml` (расширение .yaml):

```yaml
image: frontend
module: my-service
path: src
```

В Svacer появятся два проекта: **`my-service/backend`** и **`my-service/frontend`**.

## Переменные

### Обязательные (для шаблона с Vault)

В шаблоне используются секреты из Vault. В проекте должны быть заданы:

| Переменная   | Описание |
|-------------|----------|
| `VAULT_ADDR` | URL сервиса Vault (Seguro). |
| `VAULT_ROLE` | Имя роли для аутентификации в Vault. |

Секреты Svacer (`SVACER_URL`, `SVACER_IMPORT_USER`, `SVACER_IMPORT_PASSWORD`) в текущем шаблоне подставляются из Vault через переменные джобы; при необходимости их можно переопределить.

### Опциональные

| Переменная | Описание | По умолчанию |
|------------|----------|--------------|
| `SVACER_URL` | URL сервера Svacer. | из Vault |
| `SVACER_IMPORT_USER` | Пользователь для импорта в Svacer. | из Vault |
| `SVACER_IMPORT_PASSWORD` | Пароль для импорта. | из Vault |
| `PROJECT_GROUP` | Группа проектов в Svacer. | `CI_PROJECT_NAME` |
| `IF_NO_GROUP` | Действие, если группы нет. | `add` |
| `COMMIT_REF_NAME` | Имя ветки. | `CI_COMMIT_REF_NAME` |
| `COMMIT_HASH` | Хеш коммита. | `CI_COMMIT_SHA` |

## Как подключить

1. **Добавить stage** (если ещё нет):

   ```yaml
   stages:
     - static_analysis
   ```

2. **Подключить шаблон** в `.gitlab-ci.yml`:

   ```yaml
   include:
     - remote: 'https://raw.githubusercontent.com/deckhouse/modules-gitlab-ci/refs/heads/main/templates/Semgrep.gitlab-ci.yml'
   ```

   Либо через `project`/`file`, если шаблон раздаётся из вашего GitLab:

   ```yaml
   include:
    - project: deckhouse/3p/deckhouse/modules-gitlab-ci
      ref: main
      file: /templates/Semgrep.gitlab-ci.yml
   ```

3. **Задать CI/CD переменные проекта:**

   - `VAULT_ADDR` — URL Vault (Seguro).
   - `VAULT_ROLE` — роль для доступа к секретам Svacer.

   При использовании шаблона с `vault:` переменными доступ к секретам Svacer настраивается через эти две переменные; маскировать их при необходимости.

4. **Добавить конфиги Semgrep** — в каждой директории с кодом, который нужно анализировать, положить один файл `semgrep.yml` или `semgrep.yaml` (см. раздел «Формат конфига» выше). Без хотя бы одной такой директории джоба завершится с ошибкой.

Джоба из шаблона (`semgrep:scan`) не требует переопределения.

## Артефакты и коды выхода

- **Артефакты:** сохраняются в каталоге `artifacts` (отчёты SARIF и архивы для Svacer).
- **Exit code 11** — Semgrep нашёл срабатывания (findings); джоба помечена как `allow_failure` с этим кодом, пайплайн не падает.

> На данный момент ошибки при нахождении дефектов отключены.

## Краткий чеклист для внедрения

1. В репо есть хотя бы одна директория с файлом `semgrep.yml` или `semgrep.yaml` (поля `image`, `module`, `path`); в одной директории — только один из двух файлов.
2. В проекте заданы `VAULT_ADDR` и `VAULT_ROLE`.
3. В `.gitlab-ci.yml` добавлены stage `static_analysis` и `include` шаблона `Semgrep.gitlab-ci.yml`.
4. После этого джоба будет запускаться по MR, на ветках/тегах, по расписанию и вручную согласно правилам шаблона.
